The Problem

Two extremes, both broken.

Today, Filipino medical records sit on one of two ends — neither of which respects both privacy and verifiability.

Centralized EMR

  • ✗One database breach exposes every patient at once
  • ✗Hospitals own your records, not you
  • ✗No mathematical proof against tampering
  • ✗Consent is policy-based, not enforced cryptographically

Public Blockchain EMR

  • ✗Every diagnosis is publicly visible forever
  • ✗Cannot delete — violates RA 10173 §16 (Right to Erasure)
  • ✗₱344 in gas fees per single record
  • ✗Bandwidth and latency unfit for clinical use
Gen C must be private AND verifiable AND affordable AND deletable — all at once.

Algorithm 1 · The Lockbox

AES-256-GCM — the strongest standard encryption on earth.

What happens to one record
↓
Encrypted with 256-bit AES key
↓
0xA7F2E9C1B4D8F3E6...
9C2E4D1B8A6F5E2D3C1B...

Think of it like a bank vault.

🔐
256-bit key. There are more possible combinations than atoms in the observable universe. No computer today — or in the next 50 years — can guess it.
🏦
Same algorithm that protects HTTPS, your bank app, WhatsApp messages, and every government TLS connection.
🚨
GCM = "tamper alarm." The Galois Counter Mode detects if anyone changes even a single byte. Decryption fails — tampering is exposed.
🌐
Browser-side encryption. The plaintext never leaves the patient's device. Even our server never sees the unencrypted record.

Algorithm 1 · The Smart Lock

Policy-Based Attribute Encryption — who gets the key.

AES alone has one weakness: whoever holds the key can decrypt. PBAE attaches a rule directly to the encryption — only people matching the rule can unlock.

Step 1 — request
🩺 Doctor wants to read patient's X-ray
↓
Step 2 — policy stored alongside encrypted record
( Role = Doctor AND Department = Cardiology )
OR
Owner = Patient_Wallet_Address
↓
✓ MATCH
AES key released
Decryption proceeds in browser
✗ NO MATCH
HTTP 403 Forbidden
Even admins cannot override

Algorithm 1 in Motion

The hybrid encryption pipeline — five steps, all client-side.

STEP 01
🎲
Generate AES key
256-bit key generated by the patient's browser using crypto.getRandomValues. Server never sees it.
STEP 02
🔒
Encrypt record
Medical file sealed with AES-256-GCM in the browser before any network call.
STEP 03
🔑
Wrap key in PBAE policy
AES key bound to access rule — released only on policy match.
STEP 04
🌐
Upload to IPFS
Encrypted ciphertext pinned via Pinata. Returns a content address (CID).
STEP 05
🧬
Hand off to LPA
CID handed to the second algorithm for batch anchoring.
Plain medical data never touches the database, the network, or the ledger.
Only encrypted ciphertext is stored — anywhere.

Algorithm 2 · How It Works

The Merkle tree — one root proves them all.

🔒Only the root is anchored on the ledger — one transaction.
🧾Each record can prove it belongs to the batch via inclusion proof.
🔍Tamper detection: change any record → root changes.
♾️Unlimited scale — same fee for 4 or 4,000 records.
