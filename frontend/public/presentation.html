slide 04
Privacy-by-Design pillars (RA 10173 §20)
  • Encryption — AES-256-GCM at rest, performed client-side
  • Access Control — CP-ABE attribute policy: (Role:Doctor AND Dept:X) OR Owner:patient
  • Decentralization — IPFS storage + Merkle anchoring (no single point of trust)
  • Auditability — Every read/write cryptographically signed (ECDSA secp256k1)

slide 06
Hybrid encryption pipeline — 5 stages, client-side only
  • 01 Generate AES key in browser (Web Crypto API, 256 bits)
  • 02 Encrypt PDF with AES-256-GCM (authenticated, randomized IV)
  • 03 Pin ciphertext to IPFS via Pinata → returns CID
  • 04 Wrap AES key under CP-ABE attribute policy
  • 05 Enqueue CID hash into next LPA Merkle batch
slide-06-encryption-pipeline.jpeg
slide 07
Layered Proof Aggregation — anchor N records in ONE transaction
  • Without LPA = N × gas_per_tx (linear blockchain cost)
  • With LPA = 1 transaction regardless of N (batch into Merkle root)
  • At N=100 → 99.0% gas-cost reduction

slide 24
Scope limitations & boundaries
  • CP-ABE is simulated (shunting-yard policy evaluator) — real lib (OpenABE) requires PBC pairings
  • Hyperledger Besu is simulated — Merkle root + tx hash + block# generated locally
  • IPFS pinning via Pinata (centralized gateway) — true decentralization needs self-hosted IPFS node
  • No mobile app (web-only)
  • Not yet integrated with DOH FHIR profiles
