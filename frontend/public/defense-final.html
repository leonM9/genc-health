Four problems. Five objectives. One-to-one alignment.

Statement of the Problem
How can hospitals adopt a scalable, privacy-conscious, decentralized identity system?

  3. How can the LPA protocol reduce the gas cost of thousands of daily transactions?
  4. How can data be accessible only to authorized attributes, compliant with RA 10173?

Design and develop a decentralized application using AES-256, CP-ABE, and LPA.
  • SO 1. Build three modules — Registry, Patient Wallet, Provider.
  • SO 2. Implement privacy-preserving verification via Zero-Knowledge Proofs.
  • SO 3. Deploy LPA for ≥ 280 TPS at < $0.01 / record.
  • SO 4. Integrate AES-256 + CP-ABE for granular off-chain access.
  • SO 5. Evaluate against ISO/IEC 25010 via structured UAT.

Sections 3.2 – 3.3

Three layers — cleartext never leaves the browser.

Layer 1 · Presentation
React.js + WebAssembly
Client-side AES-256-GCM encryption. The medical payload is encrypted on-device before any network call. Holds the patient's DID private key.
Patient Wallet · Provider Portal · Admin Dashboard — role-scoped UIs.
Layer 2 · Logic
Node.js / FastAPI Middleware
CP-ABE authority issues attribute-based decryption keys. LPA service aggregates record hashes into Merkle batches.
SMTP multi-factor OTP — SMS rejected to defeat SIM-swap; reachable over GIDA satellite links.
Layer 3 · Data
EVM Permissioned Ledger + IPFS
QBFT consensus for immediate finality. The chain stores only the Merkle root — never the payload.
Private IPFS cluster pins encrypted ciphertext; CIDs are content-addressed and tamper-evident.
Cleartext medical data never crosses a network boundary in plaintext — privacy is enforced cryptographically, not procedurally.

Compliance 7 · LPA Time / Cost Complexity
12 / 22
Compliance Item 07 of 12

Formal bounds of our Layered Proof Aggregation protocol.

Phase 1 · Tree Construction

Time: O(n) · Cost: $0

Hashes n leaves into a binary tree off-chain. CPU-bound only. For n = 1000, completes in ≈ 38 ms on a mid-range workstation.

Phase 2 · Proof Generation

Time: O(log n) · Size: ~ 320 B

Generates an inclusion proof in logarithmic time. Proof size is exactly ⌈log₂ n⌉ × 32 bytes — sub-kilobyte even at batch 10,000.

Phase 3 · On-Chain Verify

Gas: O(log n) hash ops

The smart contract walks the proof path with keccak-256 hashes. For batch 1000, verification consumes ≈ 11 hash operations — about 25k gas, well under Polygon's 30M block limit.

Phase 4 · Cost Amortization

Cost: 1 anchor / N records

One on-chain transaction anchors the Merkle root for the entire batch — driving the marginal cost per record to 1 / N of the base fee.
